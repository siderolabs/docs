This guide explains how to install Longhorn on a Talos Linux cluster. It covers the required system extensions, kubelet mounts, disk provisioning, pod security configuration, and the final Helm installation.

## Prerequisites

Before you begin, ensure that you have the following:

- `talosctl` configured and authenticated against your cluster
- `kubectl` configured to access the same cluster
- Helm 3 installed
- At least one dedicated disk per storage node (NVMe recommended)

Longhorn also requires two Talos system extensions on every node:

- `siderolabs/iscsi-tools` — provides iscsid and iscsiadm for persistent volume operations
- `siderolabs/util-linux-tools` — provides fstrim for filesystem trimming

**If you are provisioning new nodes**, add these extensions when building your Talos image using the [Talos Image Factory UI](https://factory.talos.dev/) or during the Download installation media step when creating machines with Omni.

Select your Talos version and architecture, add both extensions (`iscsi-tools` and `util-linux-tools`) under **System Extensions**, and bootstrap your cluster using the generated image.

**If your nodes are already running without these extensions**, follow the <a href={`../../talos/${version}/platform-specific-installations/boot-assets`}>Boot Assets guide</a> for instructions on installing the required system extensions on existing nodes.

## Step 1: Configure the kubelet data path mount

Talos Linux v1.10 introduced the `/var/mnt` mount location for user-managed storage volumes.
In this guide, Longhorn will store data under `/var/mnt/longhorn`.

This path must be bind-mounted into the kubelet container so Kubernetes workloads can access it.

### 1.1: Create the kubelet mount patch

Create the following configuration patch:

```bash
cat <<EOF > longhorn-kubelet.yaml
machine:
  kubelet:
    extraMounts:
      - destination: /var/mnt/longhorn
        type: bind
        source: /var/mnt/longhorn
        options:
          - bind
          - rshared
          - rw
EOF
```

### 1.2: Apply the patch to each node

Replace `<node-ip>` with the nodes that will run Longhorn.

```bash
talosctl patch machineconfig \
  --nodes <node-ip> \
  --patch @longhorn-kubelet.yaml
```

### 1.3: Reboot the worker node

After applying the kubelet mount patch, reboot the node for the changes to take effect.

```bash
talosctl reboot --nodes <node-ip>
```
## Step 2: Provision a dedicated disk with UserVolumeConfig

Talos Linux v1.10 introduced **UserVolumeConfig**, which replaces the deprecated `machine.disks` API.

This resource automatically mounts selected disks under `/var/mnt/<name>`. In this guide, the volume is named longhorn, which mounts at `/var/mnt/longhorn`. This matches the kubelet mount configured in **Step 1**.


### 2.1: Inspect available disks

First inspect the disks available on each storage node.

```bash
talosctl get disks --nodes <node-ip>
```

Use this output to determine which disks should be used for Longhorn storage.

### 2.2: Create the UserVolumeConfig

Adjust the `diskSelector.match` and `maxsize` to match your hardware.

```bash
cat <<EOF > longhorn-user-disk.yaml
apiVersion: v1alpha1
kind: UserVolumeConfig
name: longhorn
provisioning:
  diskSelector:
    match: disk.transport == "nvme"
  grow: false
  maxSize: 10GiB
EOF
```

> **Tip:** You can target specific disks using expressions such as
> `disk.size > 50GiB` or `disk.model == "Samsung SSD 980"`.
> Use the `talosctl get disks` output to see available selector fields.

### 2.3: Apply the configuration

Replace the `<node-ip>` placeholder with the IP address of the worker node that will hold Longhorn:

```bash
talosctl patch machineconfig \
  --nodes <node-ip> \
  --patch @longhorn-user-disk.yaml
```

## Step 3: Enable privileged pod security

Longhorn requires privileged containers to manage disks and mount volumes.

Talos enables the **baseline Pod Security profile** by default, which blocks privileged workloads.

Create the Longhorn namespace with the privileged security level enabled.

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: longhorn-system
  labels:
    pod-security.kubernetes.io/enforce: privileged
EOF
```

## Step 4: Install Longhorn with Helm

Add the Longhorn Helm repository:

```bash
helm repo add longhorn https://charts.longhorn.io
helm repo update
```

Install Longhorn:

```bash
helm install longhorn longhorn/longhorn \
  --namespace longhorn-system \
  --set defaultSettings.defaultDataPath=/var/mnt/longhorn
```

This configures Longhorn to store replicas at `/var/mnt/longhorn`, which matches the disk path created in **Step 3**.

Wait for the deployment to complete:

```bash
kubectl -n longhorn-system rollout status deploy/longhorn-driver-deployer
kubectl get pods -n longhorn-system
```

## Step 5: Verify the installation

Check that the Longhorn nodes are registered:
```bash
kubectl get nodes.longhorn.io -n longhorn-system
```

You should see one entry per storage node, similar to:

```
NAME        READY   ALLOWSCHEDULING   SCHEDULABLE   AGE
worker-01   True    true              True          2m
```

Each storage node should appear with `SCHEDULABLE` set to `true` and at least one disk detected.

## Step 6: Create a test persistent volume

Create a test PersistentVolumeClaim.

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: longhorn-test-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: longhorn
  resources:
    requests:
      storage: 1Gi
EOF
```

Check its status:

```bash
kubectl get pvc longhorn-test-pvc
```

The PVC should reach the **Bound** state within a few seconds.
